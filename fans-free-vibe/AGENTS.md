# Repository Guidelines

## Overview
FANS FREE VIBE is a project for "Essays on the Effectiveness of Process-Based Vibe Coding" conducted in the software engineering course of Hongik University's Department of Computer Engineering for the first semester of 2026.

Implement the JYP FANS K-pop fan community platform that I use often. This project does not apply processes, but adopts a way of implementing them. Make sure to deploy quickly around freestyle vibration coding with minimal documentation/lightweight verification, testing.

Primary invariant: keep the free-form implementation fast while preserving FANS-like user flows and responsive mobile-first behavior.

Use pnpm and the existing Next.js/Biome/Tailwind tooling only. Do not introduce competing lockfiles, commands, or workflow cache keys.

Core stack:

- Next.js 16 App Router on React 19
- TypeScript, TSX, Tailwind CSS v4, global CSS tokens in `src/app/globals.css`
- Supabase is the planned database/auth integration; current UI work is mostly static mock implementation
- Biome for lint/format checks, Turbopack via Next.js scripts for dev/build

## Implementation Scope
This section defines which FANS service features are in scope for this assignment and which features are intentionally out of scope.

### In Scope

- Community posts and comments
- Official artist posts and replies
- Event applications
- Membership access control

### Out of Scope

- Full real-time live streaming or chat implementation
- Payment and global shipping flows
- Offline queueing, QR, or on-site operations

## Structure
```text
.
├── src/
│   ├── app/                       App Router pages, layouts, route segments, and global styles
│   │   └── [route]/_components/   Route-local components used only by that route segment
│   └── components/                Shared base/common components reused across routes
├── public/                        Static public assets generated by the starter
├── .ruler/                        Ruler-generated agent instruction source
├── .vscode/                       Workspace editor settings
├── docs/                          Not present yet; add only when documentation becomes necessary
├── scripts/                       Not present yet; add only when automation becomes necessary
└── tests/                         Not present yet; no repository-level test suite currently exists
```

Keep this file concise. Move detailed domain rules, schemas, runbooks, and platform notes into linked docs.

## Commands
```bash
pnpm install
pnpm run dev
pnpm run lint
pnpm run format
pnpm run build
# no test script is currently defined
```

## Non-Negotiables

- Preserve FANS-like navigation and mobile layout behavior.
- Prefer fast visible progress over heavy process, documentation, or exhaustive verification.
- Keep mock data and temporary UI acceptable when it helps move the free-vibe implementation forward.

## Code Quality

- Read the surrounding code before editing.
- Keep `src/app` as the App Router boundary. Put shared base components in `src/components`; put route-only components or files split only for local readability in `src/app/[route]/_components/`.
- Use `kebab-case` for file and folder names, except for framework-reserved files such as `page.tsx`, `layout.tsx`, `loading.tsx`, and `route.ts`. Use `PascalCase` for React component names.
- Prefer explicit, intention-revealing code over hidden magic.
- Keep code easy to change, even if the implementation is temporarily rough or mock-driven.
