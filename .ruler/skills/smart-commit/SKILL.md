# Smart Commit 가이드라인

## 개요

Conventional Commits 형식과 한국어 메시지로 git 커밋을 생성합니다. 스테이징된 변경사항만 커밋하는 기본 모드와, 변경사항을 논리적 단위로 자동 분할하는 스마트 멀티 커밋 모드를 지원합니다.

---

## 핵심 규칙

1. **언어**: 반드시 **한국어**로 작성. 예외 없음.
2. **형식**: Conventional Commits 형식 준수 (`type: 내용`)
3. **맥락 기반**: 실제 `git diff` 변경사항을 기반으로 작성

---

## Conventional Commits 타입

| Type       | 설명                                              |
| :--------- | :------------------------------------------------ |
| `feat`     | 새로운 기능 추가                                  |
| `fix`      | 버그 수정                                         |
| `docs`     | 문서 수정                                         |
| `style`    | 코드 포맷팅, 세미콜론 누락 등 (로직 변경 없음)    |
| `refactor` | 코드 리팩토링 (기능 변경 없음)                    |
| `perf`     | 성능 개선                                         |
| `test`     | 테스트 코드 추가 또는 수정                        |
| `build`    | 빌드 시스템, 외부 의존성 변경                     |
| `ci`       | CI/CD 설정 변경                                   |
| `chore`    | 빌드 설정, 패키지 매니저 설정 등                  |
| `revert`   | 이전 커밋 되돌리기                                |
| `init`     | 프로젝트 초기 설정                                |

---

## 커밋 메시지 가이드라인

### 기본 형식

```
type: 간결한 한국어 변경 내용
```

### 적용 범위(scope) 포함 시

```
type(scope): 간결한 한국어 변경 내용
```

### BREAKING CHANGE 시

```
feat!: 기존 API와 호환되지 않는 변경 내용
```

### 모범 사례

- 메시지는 간결하게 유지 (50자 이내 권장)
- "어떻게(how)"보다 "무엇을(what)", "왜(why)"에 집중
- 명령형 어미 사용 (예: "추가", "수정", "업데이트", "개선", "삭제")
- 최근 커밋 스타일을 참고하여 일관성 유지

### 좋은 예시 ✅

```
init: 프로젝트 초기 설정
feat: 사용자 로그인 API 추가
feat(auth): 소셜 로그인 기능 추가
fix: 결제 페이지 크래시 문제 해결
docs: README.md 설치 가이드 업데이트
style: 코드 포맷팅 (Prettier 적용)
refactor: 중복된 유틸리티 함수 제거
perf: 이미지 로딩 성능 개선
test: 결제 모듈 단위 테스트 추가
build: 의존성 패키지 업데이트
ci: GitHub Actions 배포 워크플로우 추가
chore: ESLint 설정 업데이트
revert: 사용자 로그인 API 추가 되돌리기
```

### 나쁜 예시 ❌

```
Update code        → 너무 모호함
Add login feature  → 영어로 작성됨
Login stuff        → 형식 위반
```

## 기능

### 1. 기본 모드 (스테이징된 변경사항만)

`git add`로 스테이징된 파일만 커밋합니다.

**워크플로우:**

1. `git status`로 스테이징된 변경사항 확인
2. 스테이징된 diff 검토
3. 최근 커밋 히스토리 분석 (스타일 일관성 유지)
4. 변경 유형에 맞는 type 선택
5. 간결한 한국어 커밋 메시지 작성
6. 커밋 실행

**사용 예:**
```
사용자: "커밋 만들어 줘"
어시스턴트: [스테이징된 변경사항 분석 후 단일 커밋 생성]
```

---

### 2. 스마트 커밋 모드

스테이징 여부와 관계없이 **모든 변경사항**을 분석하여, `git add -p`를 통해 논리적 단위별로 여러 개의 커밋을 생성합니다.

**워크플로우:**

1. `git status`로 전체 변경사항 확인
2. 스테이징된 diff와 미스테이징 diff 모두 검토
3. 변경사항을 논리적 단위로 분류
4. 각 단위별로:
   - `git add -p`로 해당 변경사항만 선택적 스테이징
   - 적절한 type 선택
   - 한국어 커밋 메시지 작성
   - 커밋 실행
5. 모든 변경사항이 커밋되었는지 최종 확인

**사용 예:**
```
사용자: "전체 변경사항 스마트 커밋해 줘"
사용자: "변경사항을 적절한 단위로 나눠서 커밋해 줘"
어시스턴트: [논리적으로 분리된 여러 커밋 생성]
```

**트리거 키워드:**
- "스마트 커밋"
- "변경사항 나눠서 커밋"
- "여러 커밋으로 분할"
- "적절한 단위로 커밋"

> **기본값은 기본 모드**입니다. 명시적 요청이 없으면 스마트 커밋 모드를 사용하지 않습니다.

---

## 워크플로우 결정 트리

```
커밋 요청
    │
    ├─ 스마트 커밋 명시적 요청? ────→ YES ─→ 스마트 커밋 모드
    │                                         (git add -p로 여러 커밋 생성)
    │
    └─ NO ─→ 기본 모드
              (스테이징된 변경사항으로 단일 커밋)
```

---

## 커밋 전 체크리스트

- ✓ `git status`에 커밋할 변경사항이 있는지 확인
- ✓ diff 검토를 통해 변경 내용 파악
- ✓ 최근 커밋 스타일 확인 (일관성 유지)
- ✓ 민감한 정보가 없는지 확인 (`.env`, 자격증명 등)
- ✓ 가장 적합한 type 선택
- ✓ 명확하고 간결한 한국어 메시지 작성

---

## 에러 처리

**스테이징된 변경사항 없음:**
- 스테이징된 변경사항이 없음을 사용자에게 알림
- `git add` 사용 또는 스마트 커밋 모드 요청을 제안

**머지 충돌:**
- diff에서 충돌 마커 감지
- 커밋 전에 충돌을 해결하도록 사용자에게 경고

**pre-commit 훅 실패:**
- 훅 오류 출력을 표시
- 사용자가 문제를 수정한 후 재시도
